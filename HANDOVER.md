## 1. これは何か（目的）

CTA（キャリアチケット就職）のCRO施策の勝ち/引き分け/負けを、**「なぜそうなったか（要因）」と「デザインとして何を学ぶか」まで言語化して蓄積する**ための学習サイト。

- 施策の結果数値だけでなく、**事実とAI推論（仮説）を明示的に区別**して記録する
- 各施策に「デザイン観点の活かし」（✅有効 / ⚠️注意 / ❌NG）を付け、次の施策設計に転用できる形にする
- 検証用シートPDFを受け取るたびに1施策ずつ追加していく運用

## 3. サイトの仕様

単一HTMLファイル（CDN不使用・オフライン動作）。ブラウザで開くだけで動く。

### 3層構造
1. **一覧ビュー**: 勝ち/引き分け/負けの3セクション、マソンリーカード。施策タイプは**左サイドバー**（単一選択・先頭「すべて見る」・件数バッジ付き）、デバイスチップ＋フリーテキスト検索は上部フィルタバー
   - カードはPinterest風の画像ファースト（2026-07-15リデザイン）。表示要素は **スクショサムネ（あれば）→ 勝敗バッジ＋施策タイプタグ＋ID → 施策名 → 時期 → 指標変化（最大3つ・数値のみ）** に限定。要因サマリ・デザイン観点はカードに出さない（モーダル側で表示）
   - 指標ラベルは `METRIC_LABELS` 辞書（cardHTML付近）で日本語化。新しい指標キーを使ったら辞書にも追記する
   - 指標の色は `metricClass()`: 値の文字列に「改善」があれば緑・「悪化」があれば赤を優先し、なければ+/-の符号で判定（直帰率など「下がる=良い」指標は値に「改善」を含めること）
2. **詳細モーダル**: カードクリックで開く。スクショ → 仮説 → 計測数値 → 事実vsAI推論 → 要因分析 → デザイン観点 → 次アクション
3. **横断インサイトビュー**: 全施策の学びを横断表示

### データ構造
`<script>` 冒頭の `MEASURES` 配列が全データ。1施策 = 1オブジェクト。主なフィールド:

- `id` / `title` / `type`（フィルタチップに使用）/ `pages` / `devices` / `result`（win/draw/loss）/ `adoptionLabel` / `period`
- `hypothesis` + `hypothesisIsAI`（仮説が施策ドック由来の事実か、AI推論か）
- `factuals[]`（📋事実・実線枠）/ `hypothesisInsights[]`（🤔AI推論・破線枠＋⚠️）— **この区別がサイトの核。混ぜないこと**
- `rootCause` / `metrics{}` / `nextActions[]`
- `designInsights[]`: `{type: 'good'|'caution'|'bad', text}` → ✅/⚠️/❌で表示

### スクショ埋め込み
- HTML末尾 `<!-- SCREENSHOTS-START -->`〜`<!-- SCREENSHOTS-END -->` 間の `SCREENSHOTS` オブジェクトに base64 data URIで格納
- モーダル冒頭に「📸 施策前後の比較」として表示（`SCREENSHOTS[m.id]` があれば自動表示）
- 現在スクショあり: CTA-11925 / CTA-13979

### 免責バナー
フィルタバー直下の黄色バナー。「ABテスト数値だけで判断しない」という注意書きで、「詳しく見る」で展開すると4つの外部要因（①季節性・YoY ②卒年構成の変化 ③採用市場・社会動向 ④競合・SEO変動）を表示。

## 4. 判定基準（重要）

ほとんどの施策は最終CVRで有意差が出ないため、**「案C: チームの採用判断」ベース**で分類している:

| 判定 | 基準 |
|---|---|
| ✅ 勝ち | 寄せ対応・本番採用された |
| ➖ 引き分け | デバイス別採用・検証継続中・判断保留 |
| ❌ 負け | 未採用・切り戻し |

## 6. 運用フロー: 新しい施策PDFを受け取ったら

1. **PDFを読み込む**（検証用シート。通常「【CTA】オウンド_検証用シート - {番号}：{施策名}.pdf」）
2. **判定を決める**: 採用状況の記述（寄せ対応/NA欄）から案C基準で win/draw/loss を決定。曖昧ならユーザーに確認
3. **`index.html` の `MEASURES` 配列に追加**: 事実（PDFに書いてあること）は `factuals`、データから推論したことは `hypothesisInsights` に分けて記載。`designInsights` はCTユーザー前提（スマホネイティブ・初めての就活生・情報過多で意思決定困難）で書く
4. **`docs/measures/CTA-{番号}/summary.md` を作成**（既存施策のsummary.mdの構成に倣う: 基本情報→背景→仮説→計測数値→検定→事実→AI推論→要因分析→デザイン観点→次アクション→リンク）
5. **GitHubへ反映**: `index.html` と `summary.md`（→ `measures/CTA-{番号}.md`）をリポジトリにコピー、README.mdの収録施策表を更新、commit & push

### スクショを追加する場合
1. ユーザーに前後比較画像をファイルとして保存してもらう（**チャット貼り付け画像は直接アクセス不可**）
2. `docs/measures/CTA-{番号}/before-after.png` に配置
3. Pythonスクリプトで base64 化して `SCREENSHOTS` ブロックを**全件再生成**して差し替え（マーカー間を置換する方式。過去のBashログにスクリプトあり — マーカー `<!-- SCREENSHOTS-START/END -->` 間を書き換えるだけ）
4. リポジトリの `assets/CTA-{番号}-before-after.png` にも原本を保存して push

## 7. 既知の注意点・ハマりどころ

- **CTA-13979の施策No.表記ゆれ**: PDFファイル名・Figma URLは「13979」、施策ドック本文の施策No.欄は「13917」。**13979を正式番号として扱う**（summary.mdに注記済み）
- **CTA-12699のデータ乖離**: 考察テキストは「スクロール到達率17.8%→23.3%(+5.4pt)」だが、参考データ表は「17.82%→15.11%（低下）」。別の計測定義の可能性があり**要確認**のまま（summary.md ※1参照）
- **ユーザーのファイル保存場所の罠**: パス指定で保存を依頼すると、macOSの制約でコロン区切りのフォルダ名（例: `CTA　デザイン:docs:measures:...`）として `~/Downloads/アーカイブ/` 配下に保存されることがある。見つからないときは `find ~/Downloads ~/Desktop -newermt` で当日の画像を探す
- **事実とAI推論の区別を崩さない**: PDFに書かれていない解釈を `factuals` に入れない。これはCLAUDE.mdの「事実と仮説を区別して明示する」方針の実装
- **base64埋め込みでファイルサイズが増える**: 現在約630KB。スクショが10枚を超えたあたりで、GitHub Pages + 外部画像参照方式への切り替えを検討する
- **複数セッションでの同時編集に注意**: 過去にリポジトリ側の `index.html` だけにCTA-9270が追加され、ローカル正と食い違った実例あり（2026-07-14に逆同期で解消）。**作業開始時に `git log` とローカルの差分を確認**し、リポジトリ側が進んでいたら先にローカルへ逆同期すること
