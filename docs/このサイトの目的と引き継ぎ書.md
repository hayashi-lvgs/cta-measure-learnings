## 0. このサイトの目的（最上位の狙い）

**勝ち施策・負け施策から「なぜ勝ったか / なぜ負けたか」の要因を言語化して蓄積し、次のデザイン提案に納得感（＝根拠・説得力）を持たせられるようにする。**

つまり、単なる施策の結果アーカイブではなく、**デザイン意思決定の"根拠ライブラリ"** である。

- 施策の結果数値そのものが目的ではない。「その結果から**デザインとして何を学び、次にどう使うか**」を言語化することが目的。
- 次に何かをデザイン提案するとき、「過去のCTA-◯◯でこう出た → だからこの設計にする」と**過去の実証で裏付けられた提案**ができる状態を目指す。
- そのために、以下の3つを崩さない：
  1. **事実（📋 実線）とAI推論・仮説（🤔 破線＋⚠️）を明示的に区別する** — 根拠として使うとき、どこまでが確かでどこからが解釈かを常に分かるようにする。
  2. **各施策に「デザイン観点の活かし」（✅有効 / ⚠️注意 / ❌NG）を付ける** — CTユーザー前提（スマホネイティブ・初めての就活生・情報過多で意思決定困難）で、次の設計に転用できる粒度で書く。
  3. **横断で学びをパターン化する** — 個別施策を超えて「引き算が効く」「デバイス別最適化」等の再利用可能な原則にまとめ、提案の武器にする。

### 判定基準（案C：チームの採用判断ベース）

ほとんどの施策は最終CVRで有意差が出ないため、数値の有意差ではなく**チームの採用判断**で分類する。

| 判定 | 基準 |
|---|---|
| ✅ 勝ち | 寄せ対応・本番採用された |
| ➖ 引き分け | デバイス別採用・検証継続中・判断保留 |
| ❌ 負け | 未採用・切り戻し |

## 3. サイトの仕様

CDN不使用・オフライン動作。**`index.html` をブラウザで開くだけで動く**（file:// で動作。データは `<script src>` で読み込むため fetch/JSON は使わない — file:// の CORS制限を回避するための設計）。

### 3層構造
1. **一覧ビュー**: 勝ち/引き分け/負けの3セクション、マソンリーカード。左サイドバー（256px・melta UI https://melta.tsubotax.com/ 参考）に **検索ボックス→施策タイプ（単一選択・先頭「すべて見る」・件数バッジ）→デバイスチップ→件数フッター** を集約。
   - カードはPinterest風の画像ファースト。表示要素は **スクショサムネ（あれば）→ 勝敗バッジ＋施策タイプタグ＋ID → 施策名 → 時期 → 指標変化（最大3つ・数値のみ）** に限定。要因サマリ・デザイン観点はカードに出さない（モーダル側で表示）
   - 指標ラベルは `METRIC_LABELS` 辞書（`assets/js/app.js` 内）で日本語化。新しい指標キーを使ったら辞書にも追記する
   - 指標の色は `metricClass()`: 値の文字列に「改善」があれば緑・「悪化」があれば赤を優先し、なければ+/-の符号で判定（直帰率など「下がる=良い」指標は値に「改善」を含めること）
2. **詳細モーダル**: カードクリックで開く。スクショ → 仮説 → 計測数値 → 事実vsAI推論 → 要因分析 → デザイン観点 → 次アクション
3. **横断インサイトビュー**: 全施策の学びを横断表示（`INSIGHTS` データ）

### データ構造
`data/measures.js` 冒頭の `MEASURES` 配列が全データ。1施策 = 1オブジェクト。主なフィールド:

- `id` / `title` / `type`（フィルタチップに使用）/ `pages` / `devices` / `result`（win/draw/loss）/ `adoptionLabel` / `period`
- `hypothesis` + `hypothesisIsAI`（仮説が施策ドック由来の事実か、AI推論か）
- `factuals[]`（📋事実・実線枠）/ `hypothesisInsights[]`（🤔AI推論・破線枠＋⚠️）— **この区別がサイトの核。混ぜないこと**
- `rootCause` / `metrics{}` / `nextActions[]`
- `designInsights[]`: `{type: 'good'|'caution'|'bad', text}` → ✅/⚠️/❌で表示
- 横断パターンは同ファイル内の `INSIGHTS` オブジェクト

### スクショ表示（base64廃止・外部ファイル参照）
- `data/measures.js` 末尾の `SCREENSHOTS` オブジェクトが「施策ID → 画像パス」の対応表。例: `"CTA-11925": "images/CTA-11925-before-after.png"`
- 実画像は `images/CTA-{番号}-before-after.png` に置く
- カードのサムネ・モーダル冒頭の「📸 施策前後の比較」は `SCREENSHOTS[m.id]` があれば自動表示（**画像があるカードだけサムネが出る**挙動）
- 現在スクショあり: CTA-11925 / CTA-13979

### 免責バナー
サイドバー右コンテンツ上部の黄色バナー。「ABテスト数値だけで判断しない」という注意書きで、「詳しく見る」で展開すると4つの外部要因（①季節性・YoY ②卒年構成の変化 ③採用市場・社会動向 ④競合・SEO変動）を表示。

## 5. 運用フロー: 新しい施策PDFを受け取ったら

1. **PDFを読み込む**（検証用シート。通常「【CTA】オウンド_検証用シート - {番号}：{施策名}.pdf」）
2. **判定を決める**: 採用状況の記述（寄せ対応/NA欄）から案C基準で win/draw/loss を決定。曖昧ならユーザーに確認
3. **`data/measures.js` の `MEASURES` 配列に追加**: 事実（PDFに書いてあること）は `factuals`、データから推論したことは `hypothesisInsights` に分けて記載。`designInsights` はCTユーザー前提で書く。新しい指標キーは `assets/js/app.js` の `METRIC_LABELS` にも追記
4. **`measures/CTA-{番号}.md` を作成**（既存のsummary構成に倣う: 基本情報→背景→仮説→計測数値→検定→事実→AI推論→要因分析→デザイン観点→次アクション→リンク）
5. **動作確認**: `index.html` をブラウザで開き、カード・モーダル・フィルタ・横断インサイトが正しく出るか確認
6. **GitHubへ反映**: README.mdの収録施策表を更新し、`git add -A && git commit && git push`

### スクショを追加する場合
1. ユーザーに前後比較画像をファイルとして保存してもらう（**チャット貼り付け画像は直接アクセス不可**）
2. `images/CTA-{番号}-before-after.png` に配置
3. `data/measures.js` の `SCREENSHOTS` に1行追加: `"CTA-{番号}": "images/CTA-{番号}-before-after.png"`
4. ブラウザで該当カードにサムネが出ることを確認して commit & push

## 6. 既知の注意点・ハマりどころ

- **file:// で動かすための制約**: データは必ず `<script src="data/measures.js">` 方式（classic script）で読む。`fetch()`／`type="module"` は file:// のCORS制限で動かないので使わない。
- **CTA-13979の施策No.表記ゆれ**: PDFファイル名・Figma URLは「13979」、施策ドック本文の施策No.欄は「13917」。**13979を正式番号として扱う**（summary.mdに注記済み）
- **CTA-12699のデータ乖離**: 考察テキストは「スクロール到達率17.8%→23.3%(+5.4pt)」だが、参考データ表は「17.82%→15.11%（低下）」。別の計測定義の可能性があり**要確認**のまま（summary.md ※1参照）
- **CTA-9270の有意差注記の矛盾**: シート内に『AvsBは有意な差あり』の注記があるが、掲載p値（すべて>0.05）・検定結果『いずれも有意差なし』と矛盾しており**要確認**。
- **ユーザーのファイル保存場所の罠**: パス指定で保存を依頼すると、macOSの制約でコロン区切りのフォルダ名として `~/Downloads/アーカイブ/` 配下に保存されることがある。見つからないときは `find ~/Downloads ~/Desktop -newermt` で当日の画像を探す
- **macOSのパス正規化（NFC/NFD）**: `CTA　デザイン`（全角スペース）等、日本語＋濁点を含むパスは bash のリテラル指定で `No such file or directory` になることがある。`cd`/glob/`find` を併用して実在エントリを拾う。
- **事実とAI推論の区別を崩さない**: PDFに書かれていない解釈を `factuals` に入れない。これはこのサイトの目的（提案の根拠として「どこまでが事実か」を明確にする）の実装。
- **複数セッションでの同時編集に注意**: 作業開始時に `git log` とローカル差分を確認し、リモートが進んでいたら先に pull すること。
